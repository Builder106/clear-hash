## 2026-05-27 — Web frontend redesigned as a forensic dossier #decision #milestone

Replaced the dark-slate + purple-gradient landing with a bone-white "case file" aesthetic that actually surfaces what the tool does (verify, examine, file). Cinnabar red and sage green carry signal (mismatch / verified); the Newsreader serif headline uses a marker-yellow highlight under the italicized phrase ("Rebuild the source."); a rotated examiner's stamp (SVG with text-along-circle, "CASE NO. CLH-26 · EXAMINED & FILED") sits top-right of the hero. The inspect form became a specimen intake: black SPECIMEN tag + red INSPECT submit. Result and error pages get a verdict card with a rotated seal (ATTESTED / NO ATT. / ERROR + status code) and a sage/amber/cinnabar left border, followed by an evidence table where hash strings render as monospace labels. Type stack via Google Fonts: Newsreader (variable italic axis for the marker phrase) + Geist body + JetBrains Mono. Mobile case-strip collapses to a 2×2 grid below 720px so the column labels stop colliding (caught in Playwright review — see `e41acad`). Dark-mode carbon-copy fallback preserved. **Why:** previous design read as generic SaaS and didn't lean into "supply-chain forensics." **How to apply:** every new template should compose the section-mark convention (§ NN · uppercase mono ttl · rule) and avoid bringing back the purple-gradient palette.

## 2026-05-26 — Tamper simulation + binary rename to `clearhash-handler` #decision

`crates/clearhash-cli` gained a `--simulate-tamper` flag with four modes (`injected-payload`, `content-swap`, `mode-flip`, `deletion`, plus `all`) and a new `crates/clearhash-sandbox/src/simulate_tamper.rs` module (332 lines) that deliberately mutates the registry-extracted tree before comparison. **Why:** demos and screenshots need a clean MISMATCH outcome without an actual malicious package — the four modes each surface a different diff category in the report. The CLI prints a loud `⚠ TAMPER SIMULATION` banner so the output can't be mistaken for a real catch. Same commit (`cda8b00`) renamed the web binary from `clearhash-web` to `clearhash-handler` so Vercel's function-routing layer can disambiguate it from the workspace crate name, and disambiguated the binary in CI config. **How to apply:** when writing demo collateral, prefer `--simulate-tamper all` for screenshots that need every diff variant visible; per-mode flags for narrower captures.

## 2026-05-21 — Auto-resolve `<ecosystem>:<name>` to latest stable version #decision

Inspect endpoint now accepts `npm:@sigstore/sign` (no `@<version>`) and back-fills the version by calling each ecosystem's metadata endpoint: `dist-tags.latest` for npm, `info.version` for PyPI, `crate.max_stable_version` for Cargo. Adapter trait gained `fn latest_version_url(&self, name: &str) -> Option<Url>` plus `fn parse_latest_version(&self, body: &[u8]) -> Result<Version, _>`. The resolved `PackageRef` is shown to the user with a "resolved to latest" pill so it's always obvious which version actually got inspected. **Why:** copy-pasting a bare package name is the natural input; failing the request on a missing `@version` is hostile UX. **How to apply:** new ecosystem adapters must implement both trait methods or explicitly return `None` from `latest_version_url` to opt out of resolution (Cargo previously returned `UnsupportedResolution`).

## 2026-05-21 — Vercel normalized project name to dashed form #incident

The deployed URL is `clear-hash.vercel.app`, not `clearhash.vercel.app` — Vercel's project-name normalizer inserts a hyphen between the camel-cased segments. The mismatch broke the README badge, the "Try it without installing" section, and the `curl` example rendered server-side on the landing page. Fix landed in `5dd65f0`; same commit picked up the `.vercel/` `.gitignore` entry from `vercel link`. **How to apply:** the canonical deployed URL is `https://clear-hash.vercel.app/`. Always copy from the Vercel dashboard rather than guessing from the project name.

## 2026-05-21 — Initial commit: ClearHash supply-chain integrity verifier #milestone

Public on GitHub. CLI fetches a package, verifies its SLSA attestation via Sigstore + Rekor, rebuilds it from the attested source commit in a Docker container, and compares the rebuilt file tree against the registry artifact. End-to-end verified against `npm:sigstore@2.3.1`. Web frontend is Axum + maud (no JS framework, no build pipeline), deployable to Vercel via `vercel_runtime` + `VercelLayer` or any Dockerfile host (Fly.io config also in tree). Workspace ships 7 crates under `crates/`: `clearhash-core` (types + hashing), `clearhash-registry` (npm/PyPI/Cargo fetch), `clearhash-provenance` (Sigstore + Rekor verify), `clearhash-ecosystems` (per-ecosystem adapters), `clearhash-sandbox` (Docker rebuild), `clearhash-cli` (binary), `clearhash-web` (Axum router + maud templates). 28 tests passing, clippy + fmt clean.

## 2026-05-21 — Web frontend stack: Axum + maud, no JS framework #decision

Server-rendered HTML via the `maud` Rust DSL — no template files, no build pipeline, no client JS beyond what `<script defer>` loads from Vercel's edge. The same `app()` factory in `crates/clearhash-web/src/lib.rs` powers both the standalone binary and the Vercel serverless function (`api/clearhash.rs`); anything added to either surface goes through `lib.rs` so the deploy targets stay in sync. **Why:** the entire web layer is a thin shell over the inspect pipeline — there's no client state to manage, no interactive components beyond the form submit. Going Rust + maud keeps the binary self-contained, eliminates a Node toolchain from the deploy story, and lets the same `InspectResult` struct render to HTML or JSON without a serializer translation layer. **How to apply:** any new view goes in `crates/clearhash-web/src/templates.rs`; new HTTP routes go in `lib.rs`'s `app()`. The `/inspect` endpoint is rate-limited at 30 req/min globally (`governor::RateLimiter<NotKeyed, _, _>`) — global, not per-IP, deliberately, so the free-tier deployment can't be turned into a free npm-attestation proxy.
